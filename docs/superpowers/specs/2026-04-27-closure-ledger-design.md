# Closure Ledger v1 — Design

**Date:** 2026-04-27
**Scope:** `bot_v2.py`, `walkforward_test.py`, `dashboard_server.py`, `Dashboard.html`, new `tools/reset.py`. Crypto bot is abandoned and out of scope.
**Status:** Design pending review.

---

## Problem

The new HRRR-ECMWF spread strategy has been live since the Apr 23 reset. The walkforward backtest predicted ~85% win rate / +605% simulated return on $1000. Live, the bot has produced ~4 days of closures, and there is no ergonomic way to ask "is the live result tracking the backtest?"

Specifically:

1. **No raw closure history.** Realized PnL is an aggregate counter in `state.json`. Per-trade detail lives only in market files, which get pruned on reset. There is no durable record of "every closure that happened, with the context that triggered it."
2. **No live-vs-backtest comparison.** The `+605%` sim figure is a number in a JSON file from a single Apr 23 walkforward run; nothing surfaces "we expected 85% win rate, we are at X% over N closures."
3. **Reset destroys test data.** The Apr 23 reset wiped market files manually. A second reset (Kelly tweak, threshold tweak) would destroy the data from the current test window. The realized-PnL backfill recovers the *aggregate* but not the per-trade detail.

The user wants enough signal to judge whether the in-flight test is going well, *while* it runs, and to retain that judgment across regime changes.

## Goals (v1)

- **Append-only closure ledger** at `data/closures.jsonl`, written on every closure, surviving resets.
- **Reset markers** in the same file so the dashboard can scope "current session."
- **Backtest baseline file** at `data/backtest_baseline.json`, generated by `walkforward_test.py --emit-baseline`, including a snapshot of the relevant config so the dashboard can warn when it goes stale.
- **Live-vs-backtest header strip** in the dashboard: aggregate live win rate / PnL alongside backtest expectation, with sample-size and staleness coloring.
- **Backfill** the ledger from existing market files on startup so the test data from Apr 23 onward is captured immediately.

## Non-Goals (v1) — deferred to v2

- Per-city / per-reason / per-day dashboard breakdowns.
- Migrating the existing aggregate equity curve off the market-file scan (the current chart stays).
- Per-position-open events in the ledger (only closures and reset markers).
- Streaming the ledger anywhere external; it is a local file.
- Crypto bot parity (still abandoned).

The four diagnostic cuts (per-city, per-reason, per-day, etc.) are useful only after the v1 header strip says "the strategy isn't tracking." They are diagnostic, not primary signal — building them now risks shipping unused infrastructure.

## Data Model

### `data/closures.jsonl` — append-only

One JSON object per line, two `type` values:

**`closure` row** (one per realized closure, written by `record_closure`):

```json
{
  "type": "closure",
  "ts": "2026-04-23T11:42:18+00:00",
  "city": "tokyo",
  "date_target": "2026-04-19",
  "market_id": "kxhighnyc-26apr19-t60.5",
  "question": "Will Tokyo high be 19-21°C on Apr 19?",
  "close_reason": "stop_loss",
  "opened_at": "2026-04-22T09:14:01+00:00",
  "entry_price": 0.120,
  "exit_price": 0.095,
  "shares": 41.67,
  "cost": 5.00,
  "pnl": -1.04,
  "bucket_low": 19,
  "bucket_high": 21,
  "forecast_src": "hrrr",
  "forecast_temp_at_entry": 20.1,
  "ecmwf_temp_at_entry": 17.8,
  "spread_at_entry": 2.3,
  "sigma_at_entry": 1.44,
  "p_at_entry": 0.6231,
  "ev_at_entry": 0.0451
}
```

`close_reason` is one of: `stop_loss`, `trailing_stop`, `take_profit`, `forecast_changed`, `resolved`. These are the five values already used in `bot_v2.py`.

**`reset` row** (written by `tools/reset.py`):

```json
{
  "type": "reset",
  "ts": "2026-04-23T11:27:00+00:00",
  "starting_balance": 1000.0,
  "note": "manual reset post spread-strategy migration"
}
```

The dashboard's "since last reset" filter selects ledger rows with `ts >= latest_reset_row.ts`.

### `data/backtest_baseline.json`

```json
{
  "generated_at": "2026-04-27T08:00:00+00:00",
```

## Code Changes

### `bot_v2.py` — `record_closure(mkt, pos)`

New function, **separate** from `apply_closure_to_state`. Both are called on adjacent lines at the four existing closure sites. The realized-PnL refactor that just landed only added two lines to `apply_closure_to_state`; reshaping its signature again would re-churn the same tests for a marginal win.

```python
def record_closure(mkt, pos):
    """Append one closure row to data/closures.jsonl."""
    row = {
        "type": "closure",
        "ts": pos.get("closed_at") or datetime.now(timezone.utc).isoformat(),
        "city": mkt.get("city"),
        "date_target": mkt.get("date"),
        "market_id": pos.get("market_id"),
        "question": pos.get("question"),
        "close_reason": pos.get("close_reason"),
        "opened_at": pos.get("opened_at"),
        "entry_price": pos.get("entry_price"),
        "exit_price": pos.get("exit_price"),
        "shares": pos.get("shares"),
        "cost": pos.get("cost"),
        "pnl": pos.get("pnl"),
        "bucket_low": pos.get("bucket_low"),
        "bucket_high": pos.get("bucket_high"),
        "forecast_src": pos.get("forecast_src"),
        "forecast_temp_at_entry": pos.get("forecast_temp"),
        "ecmwf_temp_at_entry": pos.get("ecmwf_temp"),
        "spread_at_entry": pos.get("spread_at_entry"),
        "sigma_at_entry": pos.get("sigma"),
        "p_at_entry": pos.get("p"),
        "ev_at_entry": pos.get("ev"),
    }
    with open(DATA_DIR / "closures.jsonl", "a") as f:
        f.write(json.dumps(row) + "\n")
```

Call sites (`bot_v2.py` line numbers approximate, current as of commit `e9d1315`):
- `:732` (stop_loss / trailing_stop in monitor)
- `:743` (forecast_changed)
- `:896` (resolved in final scan)
- `:1105` (take_profit / stop_loss / trailing_stop in `monitor_positions`)

At each: `record_closure(mkt, pos)` immediately before the existing `apply_closure_to_state(state, pnl)` call.

### Write order invariant

**`record_closure(mkt, pos)` MUST run before `save_state(state)`.**

Rationale: of the two failure modes between `record_closure` and `save_state`, the "extra ledger row" mode is at least observable and (rarely) reconcilable by hand — the inverse, "state advanced with no ledger row," loses the trade entirely with no signal. We accept the rare duplicate to guarantee the row is never silently missing. See the Risks table for the precise duplicate semantics. The four call sites already follow this order naturally (mutate `pos`/`state`, then `save_state` later in the loop); `record_closure` slots in immediately after `apply_closure_to_state`, both happening before the caller's `save_state`. Spec invariant stated explicitly so future refactors don't quietly invert it.

### `tools/reset.py` — explicit reset script

Replaces today's "manually delete `state.json` and market files." Steps:

1. Confirm with user (interactive prompt or `--yes` flag).
2. Append a `{"type": "reset", "ts": <now>, "starting_balance": <from config>, "note": <optional>}` row to `data/closures.jsonl` **before** any deletions.
3. Delete `data/state.json`.
4. Delete `data/markets/*.json` (matching today's manual procedure).
5. Print summary: ledger marker written, files cleared, restart bot.

Flags: `--yes` to skip the interactive confirmation, `--dry-run` to print the planned actions and exit without touching disk or appending the marker.

The ledger is **never deleted** by this script. That is the entire point of the design.

### `bot_v2.py` — `maybe_backfill_ledger()`

Separate from `maybe_backfill_realized_pnl`. Distinct guard:

```python
def maybe_backfill_ledger():
    """One-shot self-heal: backfill closures.jsonl from market files when
    the ledger is missing or empty but closed positions exist on disk."""
    path = DATA_DIR / "closures.jsonl"
    if path.exists() and path.stat().st_size > 0:
        return

    markets = load_all_markets()
    rows = []
    for mkt in markets:
        pos = mkt.get("position") or {}
        if pos.get("status") != "closed" and mkt.get("status") != "resolved":
            continue
        rows.append(_build_closure_row(mkt, pos))

    if not rows:
        return

    with open(path, "a") as f:
        for row in sorted(rows, key=lambda r: r.get("ts") or ""):
            f.write(json.dumps(row) + "\n")
    print(f"[backfill] closures.jsonl ← {len(rows)} rows from market files")
```

Called from the same startup path as `maybe_backfill_realized_pnl`.

**Best-effort guarantee.** Closures from before commit `eabdb67` (spread-strategy migration) lack `spread_at_entry`, `sigma_at_entry`, `forecast_src`, `ecmwf_temp_at_entry`. Backfill emits `null` for those fields. The dashboard's per-spread cuts (in v2) must filter to `spread_at_entry IS NOT NULL`. The v1 header strip uses only aggregate win rate / PnL, so this limitation does not affect v1 output.

### `walkforward_test.py` — `--emit-baseline`

New flag. Runs the existing walkforward + `--simulate` flow (no logic change), then writes `data/backtest_baseline.json` capturing test-set stats, simulation outcomes, and a config snapshot:

```python
parser.add_argument("--emit-baseline", action="store_true",
                    help="Write data/backtest_baseline.json")
```

When set, the existing `--simulate` and `--threshold` arguments still drive the run; on completion, write the JSON. The bot does not run this — it is a manual / scripted operation. Re-run when config changes.

### `dashboard_server.py` — manifest

Add to the manifest entries:

```json
"closures": "closures.jsonl",
"baseline": "backtest_baseline.json"
```

No other server changes.

## Dashboard

One new component (the only v1 dashboard surface): a header strip rendered above the existing realized-PnL tile.

**Inputs:** `closures.jsonl` (filtered to rows with `type == "closure"` and `ts >= latest_reset_ts`), `backtest_baseline.json`, `config.json`.

**Output:** one line.

```
Live (since Apr 23):  12W 3L  80.0%  +$48.32   |   Backtest:  85.0%  (n=412)
```

The primary comparison is **win rate** (directly comparable: a binary outcome regardless of bet sizing). Live PnL is shown as raw context, not compared to a backtest dollar figure — the simulation's `return_pct` is path-dependent on Kelly sizing and starting balance, so a per-period dollar expectation is not meaningful here.

**Coloring:**
- Win-rate cell green if live ≥ backtest, red if live trails by > 10 points, neutral otherwise.
- Whole strip dimmed and a `⚠ low sample` badge added when n_live < 10.
- A `⚠ baseline stale: <field>` badge if any field in `backtest_baseline.json.config_snapshot` differs from current `config.json` (e.g. user changed `SPREAD_THRESHOLD` after the last walkforward run).

**Staleness check** is a simple JS object diff: read both, compare the four keys named in `config_snapshot`. Cheap; no special handling for missing keys (treat missing as mismatch).

**Behavior with no baseline file:** strip shows live numbers only, with a `baseline missing — run walkforward_test.py --emit-baseline` hint.

The existing realized-PnL tile and aggregate equity curve are unchanged.

## Tests

- **`test_record_closure_writes_jsonl_row`** — call `record_closure` directly with a constructed `mkt`/`pos`; assert one line appended with the right shape and field values.
- **`test_all_four_closure_paths_call_record_closure`** — exercise each of the four closure sites end-to-end (stop_loss, forecast_changed, resolved, take_profit); assert ledger has exactly four `closure` rows in the right order.
- **`test_record_closure_runs_before_save_state`** — patch `save_state` to raise; assert ledger row is still present after the exception. Documents the write-order invariant.
- **`test_maybe_backfill_ledger_from_market_files`** — seed `data/markets/*.json` with mixed open/closed/resolved positions; run `maybe_backfill_ledger`; assert ledger contains one row per closure, sorted by `ts`.
- **`test_maybe_backfill_ledger_idempotent`** — run backfill twice; assert second run is a no-op.
- **`test_backfill_handles_missing_spread_fields`** — seed a market file from before the spread-strategy migration (no `spread_at_entry`); assert the row is written with `null` in those fields.
- **`tools/reset.py`** test: dry-run mode that returns the planned actions; assert reset marker would be appended before deletions.

Dashboard rendering is excluded — no JS test infrastructure exists. Pure data shape (the JSON the dashboard consumes) is covered by the Python tests above.

## Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Crash between `record_closure` and `save_state` | Narrow window (two adjacent calls). The ledger row is real (the position was closed in memory before the crash) but `state.json` and the market file did not save. On restart, the position re-loads as open, gets re-evaluated, and may close again — producing a second ledger row for what is effectively two close events with different exit prices. Acceptable in v1; manual reconciliation only if observed. Write-order invariant ensures we never lose the row in exchange for accepting the rare duplicate. |
| User runs `walkforward_test.py --emit-baseline` against a different `data-dir`, baseline reflects test markets | `--emit-baseline` records `source` path; spec note: production baseline runs against the production market archive only. |
| Two bots writing the same ledger file | Only `bot_v2.py` writes `data/closures.jsonl`. `crypto_bot.py` is abandoned and is not modified by this spec. If a second writer is ever introduced, switch to per-bot ledger files. |
| Ledger grows unbounded | At ~150 closures/month per the current strategy, file growth is ~50KB/month. No rotation needed in v1. |

## Out of Scope (v2 candidates)

- Per-city table: city × n × win_rate × live_pnl × backtest_win_rate × delta.
- Per-close-reason donut + table.
- Per-day cumulative PnL chart.
- Equity curve migrated off market-file scan onto ledger.
- Per-period reporting beyond "since reset" / "all time."
- Per-city baseline expectations in `backtest_baseline.json` (v1 only carries aggregate).
- Crypto bot parity.

These are the diagnostic cuts. They become useful when the v1 header strip says "the strategy isn't tracking." Build then.
